java-类详解

发表于 2018-10-10 | 分类于 编程语言 ， java
字数统计: 6k | 阅读时长 ≈ 21
java中的类

引言

java是纯面向对象的编程语言，所有类，包括我们自己编写的类，都是Object的子类

类

类是对具有相同特征的实体的抽象，是面向对象编程的核心，基于类我们才能实现封装、继承、多态等面向对象的三大特性。

封装 | 继承 | 多态

封装：将实体抽象成一个对象，并将其所具备的属性和方法封装在一起，使其构成一个不可分割的独立实体，对外隐藏内部细节，只提供必要的方法供外部使用

手机不知道里面有哪些元器件，一般也不能随意拆开，元器件就是属性，一般不直接对外暴露，但是只要手机没坏，我们就可以通过触摸屏使用手机，也就是手机对外提供的方法：响应触屏操作

继承：实现了IS-A的关系，拥有父类所有属性和方法，但是私有属性和方法无法访问，同时拥有自己的特征[属性和方法]；

猫是一种动物，猫继承了动物类，也就有了动物的属性和方法，但是猫也有自己的特征，比如叫声不一样

多态：分为运行时多态和编译时多态，编译时多态一般就是方法重载，根据方法的参数不同，执行不同的方法；运行时多态则需要实现继承、重写和向上转型，指对象引用所指向的具体类型在运行时才能确定，一般是父类的引用指向子类的对象

父类数组，存储子类对象，遍历执行时会调用子类重写的方法

重写和重载

  • 重写 ：继承关系中，覆盖原来的实现，也就是说子类实现了一个和父类完全一样的方法，需要满足访问权限子类大于父类，也就是访问权限变松，子类返回类型是父类同类型或者子类型，使用 @Override 注解
  • 重载：一个类中参数列表不同的同名方法，参数列表不同指的是参数类型、个数、顺序至少有一项不同，返回值不同不算重载，无法通过编译，因为不知道要调用哪个方法

类的访问修饰符

访问修饰符可以修饰类的属性和方法，其目的在于保证封装这个特性

访问修饰符 同类 同包 子类(不同包) 不同包(其他类)
public 允许 允许 允许 允许
protected 允许 允许 允许 不允许
default 允许 允许 不允许 不允许
private 允许 不允许 不允许 不允许

父类和子类的访问权限关系

  • 子类访问权限比父类更松

    父类: private

    子类: public

  • 子类的方法返回值类型是父类的方法返回值类型的子类或者同类

    父类: Object

    子类: String

  • 子类抛出的异常范围<父类抛出的异常范围

    父类: Exception

    子类: IOException

类的关系

类主要有三种关系，泛化（继承）、实现（接口）、组合 | 聚合（成员变量）

泛化

泛化（继承）指的是子类继承父类（具体类 | 抽象类），表示一种IS-A的关系

子类会拥有父类的所有属性和方法，但是无法访问private的方法

就像你虽然继承了家族遗产，但是没有给你密码，你也用不了

泛化的实现有两种

  1. extends，直接继承具体父类或者抽象父类
  2. 组合的方式，父类对象作为子类的成员变量（其实不能确定是不是一种泛化的实现，有点偏代理的感觉）

java中类加载的双亲委派机制，子类加载器和父类加载器就是以组合的方式实现类继承效果的，子类加载器在加载类时，会先委托给父类加载器进行加载

实现

实现指的是类实现接口，具备接口所规范的某种功能

类实现接口，会拥有接口所定义的方法、属性

需要注意接口中

  1. 方法默认是public，但是也允许private进行修饰

    需要显式声明，已声明则必须实现

  2. 属性默认是public static final，并且不允许修改

    默认的意思是，即便不加修饰符，也会在编译时自动补全

  3. 方法可不实现，也可以有实现[需要用default修饰，之前只能抽象方法]

实际上，接口的规范是在JDK8和9时新增了默认实现、私有方法和静态方法

私有方法只能接口内部调用，有什么作用呢？

比如接口提供了3个方法的默认实现，3个方法有重复代码段，首先想到的就是将其抽象为一个方法，但是该方法默认是public，实现该接口的类就能直接使用该方法了，安全性遭到破坏，所以就引入了private的default方法

组合

一般是以成员变量的形式实现的，组合的作用很多

  1. 实现继承关系，比如java类加载器的双亲委托机制
  2. 代理模式，比如静态代理显式持有被代理对象，通过被代理对象调用原本的方法；JDK动态代理是动态生成被代理接口的实现类，该类持有调用处理器执行附加的额外逻辑
  3. 装饰器模式，比如MyBatis中，CacheingExecutor持有一个名为delegate的成员变量，delegate是基本的三种执行器，通过delegate能够调用原始执行器的相关方法

构造函数

构造函数的作用就是执行类的初始化工作，名字和类名一致，无返回值，但是不能用void声明，new的时候自动调用，当然也可以通过反射机制获取类的构造函数，用newInstance方法调用

没有构造函数会有问题吗

没问题，执行默认的无参构造函数[自动生成的]，但是如果我们自己实现了构造函数，那么就不会有默认的无参构造函数，此时new对象如果不传参就会报错

定义无参、不做事的构造函数的原因

在构造子类对象时，如果没有显式地用super调用父类的构造方法，则会默认执行父类的无参构造方法，而如果父类没有无参构造方法[有可能存在自动生成的默认构造函数]，则会报错，因此可以在父类定义一个不做事且无参的构造方法[已经定义了有参构造函数，额外定义一个无参、不做事的构造函数]

构造器链是什么

当你在一个构造函数中调用另一个构造函数，就是构造器链。这种情况只在重载了类的构造器的时候才会出现

比如 一个类是用户 包括 设置用户名、ID、密码、头像的构造函数 A和一个设置用户名、ID、密码、头像、性别的构造函数B

B的代码中可以调用构造函数A，避免写太多的this.x=x

类生命周期

类生命周期一般包括：加载→连接:[验证→准备→解析]→初始化→[使用→卸载]，其中加载到初始化，统称为类加载过程

虽然生命周期包括上述阶段，但是实际上为了支持java语言的动态性，解析阶段有可能是在初始化之后才开始的

类加载目标

类加载，就是把类对应的.class文件加载到内存中（类的元数据、类的静态变量、类的Class对象），方便我们后续使用

  1. 安全检查
  2. 把.class文件扔到内存中，要不然使用的时候找不到[类的元数据]
  3. 提供程序员访问类信息的入口[class对象]
  4. 分配内存，存储类的静态变量，设置静态变量的值[静态变量]
  5. 确定类的方法怎么调用，静态变量怎么访问[类的元数据]

详细过程

  1. 加载阶段：通过类全名找到.class文件，将其转换为运行时数据结构[常量池、类字段、类方法]，装载到元空间中，并在堆中生成一个class对象[只是一个指针，指向元空间的类元数据]作为访问入口[getClass() | Class.forname()]

  2. 验证阶段：被加载的类是否规范，是否满足虚拟机安全性的约束

  3. 准备阶段：为类静态变量分配内存，并依据类型设置初始值，比如int为0，boolean为false

    static final修饰的直接赋值为具体值

  4. 解析阶段：将常量池内的符号引用替换为直接引用[指针或者偏移量]，方便后续调用

    程序执行方法时，需要知道方法在哪里可以调用，虚拟机为每个类准备了一个方法表，只要知道方法的偏移量就能定位调用到这个方法，解析就是将符号引用替换为在方法表中的位置的过程

  5. 初始化阶段：真正执行类中定义的Java程序代码[实际上是字节码]，到这里才会给静态变量赋我们给的初始值，并且通过锁的机制保证只初始化一次

    什么时候会触发类的初始化呢？就是看啥时候需要访问类的静态变量或者调用静态方法

    1. new一个类的实例
    2. 访问类的静态变量
    3. 调用类的静态方法
    4. 初始化A类时，如果发现A类的父类没有初始化，就得先初始化父类
    5. 通过反射调用类

类的卸载

首先，类卸载意味着类的class对象，类的元数据都被垃圾回收，需要满足以下条件：

  1. 所有实例对象都已被回收，堆中没有任何类的对象或者子类对象
  2. 该类对应的Class对象没有在其他地方被引用，无法通过反射访问该类
  3. 该类的类加载器已经被回收

一般不会发生类卸载，因为虚拟机的类加载器[BootstrapClassLorder | ExtesionClassLorder | AppClassLorder]不会被回收（但是我们自定义的类加载器加载的类可能会被回收），所以不满足类卸载的第3个条件，自然不会卸载

类加载器

定义

类加载器就是专门负责加载类的类，主要方法是findClass()和loadClass()，需要将.class文件装载到虚拟机，在堆中生成一个代表该类的从class对象。

类加载策略

  1. 全盘负责：负责加载该类的类加载器默认会一并负责加载该类所依赖和引用的其它类

  2. 缓存机制：保证所有加载过的类不会重复加载，加载前先去缓存中看能不能找到这个类，找不到才去加载

  3. 双亲委派机制：如果一个类加载器收到了类加载的请求，它不会自己去尝试加载这个类，而是把请求委托给父加载器去完成，依次向上，因此，所有的类加载请求最终都应该被传递到顶层的启动类加载器中，只有当父加载器在它的搜索范围中没有找到所需的类时，即无法完成该加载，子加载器才会尝试自己去加载该类

    比如我们自己编写一个Objec类，程序运行时，就会有两个Object，双亲委派能够确保加载的是原生的Object。这是因为加载时，首先找AppClassLorder，AppClassLorder找ExtensionClasslorder，最后找到BootstrapClassLoader进行加载，它发现Object已经加载过了，就不会加载了

  4. 按需加载：虚拟机启动的时候不会一次性加载所有类，而是按需动态加载，也就是在具体用到的时候再加载

内置类加载器

  1. BootstrapClasslorder(启动类加载器)：最顶层的加载类，getParent()获取到的是null[C++实现，自然没有父类]，主要用来加载JDK内部的核心类库
  2. ExtensionClassLoader(扩展类加载器)：主要负责加载 JDK扩展目录下的jar包，父类是BootstrapClasslorder
  3. AppClassLoader(应用程序类加载器)：面向用户的加载器，负责加载当前应用 classpath 下的所有 jar 包和类，其实就是用户自定义的类，父类是ExtensionClassLoader

自定义类加载器

比如我们可以对.java文件进行加密，类加载器先要解密才能加载

自定义首先需要继承classlorder这个类，有两个方法可以重写

  1. findClass():根据类的二进制名称来查找类
  2. loadClass():加载指定二进制名称的类，实现了双亲委派机制

一般来说，只需要重写findClass方法即可，如果需要打破双亲委派机制，那么可以重写loadClass方法，没事不要打破

Object类

Object是java所有类的父类，拥有很多原生方法，按照对象操作、线程操作等可以进行划分

equals 对比 ==

  1. ==是操作符，equals是Object的原生方法

  2. 判断对象是否相等的逻辑不同

    1. 基本数据类型，只能用==判断两个值是否相等，无法调用equals方法

    2. 引用类型，==判断是否引用了同一个对象，比较的是内存地址，equals是判断引用的对象是否等价，如果没有重写equals方法，等价于==判断

      一般需要重写该方法，根据依据内容是否相等来判断，比如Integer类就重写了该方法，依据值进行比较

      1. 同一个对象的引用，返回true
      2. 不同类型 或者 入参为空，返回false
      3. 进行类型转换，判断每个字段是否相等，全部相等返回true

equals 和 hashCode

hashCode返回一个整型的哈希值，用于计算对象在哈希表中的索引位置，主要作用在于寻址，减少查询时间而不是用来判断是否相等的

比如，在list中如果要判断元素是否存在，需要遍历，但是用hashset可以快速定位到桶[链表 | 红黑树]的位置，这个桶里如果都没有，那么不可能存在

上述实现可行的原因在于：对象相等→哈希值相等→位于同一个桶

hashCode基于内容计算哈希值的，虽然相等的概率很小，但是也是有可能的

为什么两个不同对象的哈希值

实际上是因为不管我们的计算方式多么复杂，取值范围依然是无限大的，但是哈希值的范围只是一个整型，所以肯定会存在哈希冲突，下面是两个String字符串拥有相同的哈希码

equals判断两个对象是否等价，等价的对象哈希值必定相等，但是哈希值相同的两个对象，不一定等价，也就是所谓的哈希冲突

重写equals方法，需要重写hashCode()方法，保证哈希值一样

比如业务逻辑上，我们重写equals方法实现“如果学生的所有字段的值都一致，那就是同一个对象”，但是没有重写hashCode方法：假设此时，我们需要将学生置于HashSet中进行注册信息的去重，那么因为HashSet是先用哈希值计算下标，再调用equals比较是否相等，而原生hashCode方法基于内存地址计算哈希值，自然不相等，那就无法实现去重的功能了

clone

clone方法是Object的protected方法，负责深拷贝（新对象），无法直接调用。

  1. 为什么不能访问protected方法呢？protected不是允许不同包的子类调用吗？

    因为protected所允许的不同包的子类调用是指 在子类中可以访问继承来的成员，用该成员可以访问protected方法

  2. 如何实现深拷贝呢？

    需要实现cloneable接口然后重写clone方法，但是clone方法不是这个接口定义的，而是Object类中的一个protected方法，cloneable只是一个规定，这种方法复杂还有风险，一般使用拷贝构造函数或者拷贝工厂来实现深拷贝，比如对于一个类，拷贝构造函数就是实例化一个类对象，按照类原本的字段值进行赋值，然后返回

hashCode()的乘数31

以String对象的哈希计算为例：s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]

每个值都需要和31相乘，为什么选择31呢？

实际上需要考虑

  1. 哈希冲突：太小哈希冲突比较严重，质数难以整除任何数，也不会被任何数整除，哈希冲突概率很小，下标分布更为均匀
  2. 信息保留：取值太大，哈希值可能超过int型的表示范围，导致丢失信息
  3. 优化：任意数与$2^n$相乘，可以用左移操作进行优化$31×i=(32-1)×i=2^5×i-i=(i<<5)-i$，因为不能直接用偶数，所以选择差值为1的质数

综上所述，选择31

关键字

this

this是指向自己的指针，一般用于

  1. 避免属性和入参重名
  2. 调用另外一个构造函数，形成构造器链

super

  • super()访问父类构造函数，委托父类进行初始化工作

  • super.func()访问父类的方法，比如子类重写了某个方法，用super可以访问父类的实现

  • super.x访问父类的非私有属性，私有属性无法访问

final

数据：final将数据声明为不可变的。基本类型，值不可变；引用类型，不能引用其他对象[但被引用的对象的内容可修改]

方法：该方法不能被子类重写，如果有相同签名的方法，实际上是新的方法，private的方法隐式指定为final

类：类不允许被继承

static

静态变量：该变量是属于类的，而不属于类对象，类所有实例共享静态变量，可以直接通过类名进行访问，内存中只存在一份，在类加载的准备阶段为静态变量准备内存空间，并赋予基于类型的初始值，然后在初始化阶段修改为真正的初始值

静态方法：也是属于类的，因此必须实现，并且在类没有实例化前就能调用，因此只能访问静态变量和调用静态方法

静态语句块：类初始化时运行一次，先于构造方法执行，主要用于一次性地对类进行初始化设置

静态内部类：非静态内部类依赖于外部类的实例，而静态内部类不需要，因此静态内部类不能访问外部类的非静态的常量和方法

上述概念都需要用static进行声明

初始化顺序：

  1. 静态变量和静态代码块的执行顺序取决于代码顺序
  2. 实例变量和普通语句块
  3. 构造函数

优点

  1. 属于类，不需要创建类对象就可以访问
  2. 全局唯一，可以用来表示全局状态
  3. 不同线程可以共享

缺点

  1. 不会自动摧毁
  2. 多线程需要考虑并发问题

拓展问题

数组类是如何创建的？

数组类不是由classlorder创建的，而是虚拟机在需要的时候自动创建的，而数组类通过getClasslorder方法获取到的类加载器和该数组的元素类型的类加载器一致

如何判断两个类是否是一样的

不仅仅依靠类的全名，还要看类的加载器是否一致

什么叫类优先原则

子类（实现类）继承了父类并实现了接口；父类和接口中声明了同名同参数的方法。

默认情况下，子类（实现类）在没有重写此方法的情况下，调用的是父类的方法，而不是接口中的方法，通过接口名.super.方法名，可以调用接口的方法

抽象类和接口的区别

关键字：接口用interface来定义，用implements来实现；抽象类用abstract定义，用extends来继承

类关系：接口允许多实现，抽象类只能单继承

修饰符：接口方法的修饰符默认为public[也可以private]，抽象方法不能为private，因为本来就是给子类去实现的

成员变量：抽象类可以有普通成员变量，起到占位符的作用，相当于约定有这么个属性，接口的成员变量只能是public static final

作用：抽象类是IS-A的关系，比如狗是一种动物，是对公共属性和方法的抽取；接口则是对能力的抽取，也可以作为一种规范和约束

为什么要使用接口

主要是因为类只能单继承，接口允许多实现，使用起来更加方便，比如JDK动态代理就是基于接口的实现类定义的

什么是不可变对象(immutable object)? Java 中怎么创建一个不可变对象?

不可变就是指对象一旦被创建，状态不能被改变，任何修改实际上都是创建一个新的对象，比如String、Integer或者其他包装类

怎么创建呢？需要满足以下原则

  1. 类的所有属性是final的，并且是private的
  2. 类是final的，限制子类继承父类改变父类的不可变性质
  3. 类不提供对属性的set方法供外部使用
  4. 类的字段如果需要访问，则提供字段的深拷贝
