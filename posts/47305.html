java中的字符串String

引言

String在使用时经常会认为是一种基本数据类型，但实际上String是引用类型，并且比较特殊，可以通过“”或者new的方式创建

字符串和字符

  1. 形式上：字符是单引号引起来的单个字符，字符串则是双引号引起来的若干个字符
  2. 含义上：字符相当于ASCII值对应的整数，可以参与表达式运算，比如我们经常用到的字符-‘a’得到字符在数组中从0开始的下标；字符串则表示字符串在内存中的存储地址
  3. 占据内存大小：字符占据固定2个字节的内存空间，字符串根据字符串长度占据若干字节[String的底层存储改为了byte]

不可变

如何保证String的不可变

  1. String类被final修饰，不能被继承，无法通过继承破坏其不变性
  2. 内部使用char | byte数组存储数据，数组也是final的，无法改变指向
  3. String类没有提供任何方法能够获取数组或者修改数组内容

但是，实际上通过反射机制是可以破坏String的不变性的

为什么需要保证String的不可变

  1. 安全性：String类型是频繁使用的类型，经常会用于存储一些敏感信息，比如用户名、密码等等，如果String可变，那么攻击者可能先使用合法的数据通过安全检查，然后修改对象进行一些危险操作
  2. 节约内存消耗：只要一个String对象创建过了[引号]，那么内容相同的字符串可以使用字符串常量池中的同一个对象，避免创建多个重复字符串对象，如果String对象可变，那么一个地方变，就会导致其他地方也会发生变化，那就无法支持常量池机制了
  3. 线程安全：String的不可变使其天生具备线程安全的性质，如果线程尝试修改String，则会创建一个新的String对象，不存在并发修改的问题，因此可以在不同的线程中使用
  4. 节省计算开销：哈希表中一般都用String作为key，而用String作为key，哈希值只需要计算一次就可以缓存起来，下次可以直接拿缓存值来计算下标

数据结构

数据结构其实就是一个数组，只是数组类型由char[]改为了byte[]

JDK9为什么要将String的底层实现由char[]改为byte[]

这是因为每个字符char由2个字节组成，而开发环境中经常使用的是Latin1编码中的字符，只需要一个字节就可以存储，如果用char数组来存储就有一半的存储空间被浪费了，如果用byte数组来存储就不会浪费。但是，对于一些不属于Latin1编码的字符，比如中文，String同样会将其转换为字节存储到字节数组中，只是编码不是Latin1，而是UTF16

字符串常量池

字符串常量池和缓冲池差不多，存储了两种对象

  1. 字符串
  2. 堆中字符串对象的引用

在JDK1.8之后，包括1.8，字符串常量池逻辑上存储在虚拟机的方法区[元空间和堆共同实现]中，物理上存储在堆中

String StringBuffer StringBuilder

对比

  1. 不可变性

    String不可变，我们修改String会产生一个新的String对象，然后将引用指向它

    StringBuffer和StringBuilder可变

  2. 线程安全

    String 不可变，天生是线程安全的

    StringBuffer 是线程安全的，内部使用 synchronized 进行同步

    StringBuilder 不是线程安全的

  3. 应用场景

    字符串一般不会发生变化，用String即可

    如果业务逻辑涉及到大量对字符串的修改，比如拼接字符串，但是只需要单线程，那么用StringBuilder

    如果需要并发访问，则需要使用StringBuffer，但是运行速度会变慢，因为会检查是否拥有锁

StringBuilder和StringBuffer的底层实现

这两个实际上都是AbstractStringBuilder的子类，存储了一个byte类型的数组，所以添加元素时可能会扩容，扩容的大小默认是2倍+2，然后使用Arrays.cppyOf方法进行复制

比较重要的还有一个toString方法返回一个String，内部实际上就是根据编码类型，是Latin1还是UTF16，调用newString()方法，最后调用的实际上是String接收byte[]数组的构造函数，创建了一个堆中的String对象

StringBuffer相对于StringBuilder就是存在并发控制的方法添加了synchronized关键字实现线程安全，比如扩容方法，trimToSize方法

intern()

该方法是String对象的方法，用于保证相同内容的字符串常量引用同一个常量池中的对象

如果字符串常量池已经包含一个等于(通过equals方法比较)此String对象的字符串，则返回字符串常量池中这个字符串的引用, 否则将当前String对象的引用地址（堆中对象的引用地址）添加（或者叫复制）到字符串常量池中并返回，这么做是为了节约堆空间，毕竟都在堆中。

比如我们用new的方法创建两个内容相同的String对象，那么他们是不相等的，而如果我们调用intern()返回的字符串对象则和原来的字符串对象是一个对象

操作优化

  1. 如果拼接的字符串都是直接量，编译器会优化为一个完整的字符串

    String s1 = "a" + "b"，编译器会优化为String s1 = "ab"，所以不会在常量池中尝试创建”a”和”b”

  2. 如果存在变量，会使用StringBuilder进行优化，创建一个StringBuilder对象，调用其append方法

    既然已经使用StringBuilder进行优化了，为什么多次拼接操作时，直接使用StringBuilder比使用String快？

    因为每次进行+=操作，虽然会进行优化，但是都得创建一个StringBuilder对象，调用append方法，再toString返回String对象

语句分析

intern()

  1. 首先尝试创建“1”，最后s3是一个堆中的String对象，值是11

  2. 在字符串常量池中没有11，就会创建一个指向堆中s3的引用，代表字符串11

  3. s4指向字符串常量池中的11，其实就是指向堆中的s3

  4. 自然s3和s4相等

s3和s4不相等

java原生字符串常量

String创建了多少对象

  1. "a" "b"会尝试在字符串常量池中创建对象，最多创建2个
  2. new String一个a和b，2个
  3. 涉及到String对象的拼接，需要用到StringBuilder，1个
  4. StringBuilder的toString()方法会新建一个String，并且没有使用引号（byte数组），不会尝试在字符串常量池中创建对象，1个
  5. intern(): 字符串常量池如果有"ab"，就返回这个ab，否则返回堆中"ab"对象的引用，不管咋样，都没有创建对象

总共创建[4,6]个对象

  1. 没有进行优化，所以"A" | "B" | "AB"： 如果字符串常量池都没有，最多创建三个

  2. new创建1个
