DMA-数据库设计

发表于 2019-01-24 | 分类于 课程学习 ， 数据库系统及其应用
字数统计: 2k | 阅读时长 ≈ 6
数据库设计相关概念，基于东南大学徐立臻数据库课程

数据依赖

  • 关系数据库内数据之间存在一定的数据依赖关系

函数依赖FD

Function Dependency

一个属性的值可以唯一的决定其它属性的值

学号可以唯一决定姓名、年龄等等

多值依赖MVD

Multi-valued Dependency

一个属性的值决定其它属性的一组值

实际生活存在但很少

连接依赖JD

Join Dependency

关系属性之间能够无损连接

  • 无损连接：连接后的元组数一个不多，一个不少

实际生活存在但很少

供应商-零件-工程的三元关系

某一个供应商供应零件某个操作

分解成三张表：供应商-零件 供应商-工程 零件-工程

再把这三张表连接能够完全还原原来的表

那么就叫无损连接分解

例如 某工程需要某零件，并且某供应商提供该零件，那么该工程必须去采购该供应商提供的零件才能满足无损连接分解，否则还原就会出现之前没有的记录

关系模式的规范化

关系模式的范式理论

1NF

第一范式

不支持表中套表

关系中的每一个元组必须是原子【即不可再分】

分解Address

2NF

第二范式

数据库设计时，设计出来的表满足一范式，并且该表内不存在属性对主键的部分函数依赖

例子：

若属性由（学号、姓名、班级、课程号、成绩）五个属性组成，其中（学号，课程号）共同组成主键。

  • 此时不满足二范式。
  • 其中的姓名和班级，只需要依赖主键中的学号而不需要课程号就可以得到

不满足2范式容易出现的问题：

  • 插入异常

    上例中，不能插入一个还未选课的学生的信息，逐渐不能为空

  • 删除异常

    上例中，如果一个学生申请休学，把选过的课退了，那么他的信息也会被删除

  • 更新异常

    更新中难以保持数据的一致性，上例的设计有大量的数据冗余

    每次更新学生数据，每一个选课记录都得改

解决方法
  • 设计时，一事一地。
    • 一张表只管一件事情
    • 可以分解为学号-课程号-成绩和学号-姓名-班级两张表

3 NF

第三范式

在满足二范式的前提下，不存在属性对主键的传递依赖。

例子

若属性由（职工编号、工资级别、工资）三个属性组成，其中（职工编号）为主键。

分析

工资取决于工资级别，工资级别取决于职工编号

上例不满足3范式的问题
  • 插入异常：当一个人的工资级别还没定的时候，他对应的工资也没有
  • 删除异常：3级工资的人只有一个，若删除，会把对应的工资级别信息也删除了
  • 更新异常：数据内有大量的冗余，1级工资对应1000元，会重复很多次，修改也很麻烦
解决方法
  • 设计时，一事一地。
    • 一张表只管一件事情
    • 一张职工编号-职工等级表，一张职工等级-工资表

4 NF 四范式

  • 在满足三范式的前提下，消除多值依赖。

5 NF 五范式

  • 在满足四范式的前提下，消除连接依赖。

设计如下三张表：

  1. 第一行设备基本信息表
  2. 第二行设备存放位置
  3. 第三行设备入库出库的记录

ER模型

数据库设计方法

面向过程的方法

根据单位日常处理的流程，以过程为中心。

  • 好处
    • 在设计初期，能比较快的实现
  • 缺点
    • 没分析数据之间的关系，数据有冗余和矛盾，当流程进行改动时，会有很多问题

面向对象的方法

依然会分析流程，但是需要归纳流程涉及到的数据，是以数据为中心的方法

分析数据之间的关系，设计一些符合3NF的模式

数据库设计流程

  1. Requirement Analysis

    需求分析，与用户交流确定需求

    1. Information requirement

      每一项业务需要涉及到哪些信息

    2. Process requirement

      每一项业务需要对这些数据做什么处理（增删改查）

    3. Requirement indication

      结果是得到需求分析报告，需要和用户反复讨论、确认

  2. Concept Design

    概念设计，与用户讨论，分析数据之间的关系（实体）、实体及实体间的逻辑，利用ER图来进行描述，和用什么数据库系统实现无关

    得到DBMS independent data schema

    确保用户的所有业务都能在我们设计的schema基础上顺利执行

  3. Logic Design

    逻辑设计，看采用哪一种数据库系统，生成全局模式（基表的属性、属性类型等等）【concept schema】，定义视图【outer schema】

  4. Physical Design

    物理设计，考虑数据在磁盘上到底如何存储，默认都是堆文件，但是比如学生基本信息表比较适合用簇集，而在某些属性上是否应该添加B+树索引

    得到【inner schema】

需求分析

数据字典：把所有的基本数据元素都找出来

数据流图DFD or UML

解决问题：

  • 名字冲突

    • 同名异意，都叫编号，学号、设备编号啥的
    • 异名同意，学号、学生编号啥的

  • 概念冲突

    同一个概念，一个部门用作属性，一个部门用作实体

    院系，可以作为学生表的属性，也可以作为实体（表）

    是否需要视为实体，如果需要，则如何设置属性指向该表，院系编号

  • 域冲突

    同一个概念，名称、意义都没有区别，但是值域有区别，比如性别可以男\女、0\1、F\M

  • 编码

    • 实现信息的标准化，学号（专业、院系、入学年份）
    • 标识符，唯一标识实体
    • 压缩信息，院系单独作为一张表，编号代表院系，而不是全称

通过需求分析，所有信息都要有唯一的来源和职责

概念设计

在数据字典和数据流图的基础上

  • 用ER图抽象出实体，哪些数据应该抽象形成一个实体
  • 定义实体间的联系
  • 绘制ER图（ERWin，Rose），与用户讨论

逻辑设计

  • 把ER图表达的数据模型进行建表
  • 表和属性的命名规则
  • 定义每个属性的类型和值域

  • 适当的逆规范化

    • 权衡效率和范式规范
    • 采取一些存储过程、编程规范防止出现逆规范的问题

  • 定义必要的视图

  • 考虑遗留系统的接口表的设计

物理设计

根据DBMS特点对每个表的存储和索引情况进行权衡

  • 分区设计
  • 存储过程
  • 触发器
  • 完整性约束

总结

  • 仅仅在结构上满足3范式是不够的
  • 一事一地包括每项信息的唯一，要提取出问题的本质，识别本质上同一概念的信息项
  • 对于表达类似信息、能合并尽量合并
  • 考虑到效率、用途等因素、该分开的要分开
  • 结合DBMS内部实现技术，合理设计索引和文件结构，为查询优化准备号存取路径
  • 在结构规范化、减少数据冗余和提高数据库访问性能之间仔细权衡，适当折中
