Git学习

Git

Git是什么

Git是一个免费开源的分布式版本控制系统，它使用一个特殊的叫做仓库的数据库来记录文件的变化，仓库中的每个文件都有一个完整的版本历史记录，可以看到谁在什么时间修改了哪些文件的哪些内容，在需要的时候也可以将文件恢复到之前的某一个版本

版本控制系统

集中式

所有的文件都保存在中央服务器上，每个人的电脑上只保存了一个副本，当你需要修改文件的时候，首先要从中央服务器上下载最新的版本，然后添加你想要的修改内容，修改完成之后再上传回中央服务器

优势在于使用很简单直接，缺陷是中央服务器的单点故障问题：如果中央服务器出现故障或者网络连接出现问题，那么所有人都无法工作了，只能等待中央服务器或者网络恢复正常

分布式

每个人在电脑上都有一个完整的版本哭，所以我们可以在本地进行修改，不需要考虑网络问题，这样即使中央服务器出现故障，我们也可以继续工作，当需要将我们的修改内容分享给其他人的时候，只需要将仓库互相同步一下就可以了

工作区域和文件状态

Git的本地数据管理分为三个区域：

  1. 工作区：电脑上资源管理器能看到的目录、文件（.git所在的目录，可以进行操作）
  2. 暂存区：临时存储区域，用于保存即将提交到Git仓库的修改内容，用于进行版本控制（中间区域，临时存放内容）
  3. 本地仓库：存储完整的项目历史和元数据，是Git存储代码和版本信息的主要位置

首先通过git add将被修改的文件添加到暂存区，然后通过git commit将文件内容统一提交到本地仓库，在这个过程中可以使用相关命令进行查看、比较或者撤销修改

工作区的文件也分为四种状态：

  1. 未跟踪：新创建的文件，还没有被Git管理起来的文件
  2. 未修改：已被管理但文件的内容没有修改
  3. 已修改：已被管理且文件内容有所修改
  4. 已暂存：已经添加到暂存区中的文件

回退版本

git reset

退回到之前某一个提交commit的状态，有三种用法

  1. git reset --soft：回退到某一个版本，并且保留工作区和暂存区的所有修改内容
  2. git reset --hard：回退到某一个版本，但是丢弃工作区和暂存区的所有修改内容
  3. git reset --mixed（默认参数）：回退到某一个版本，并且只保留工作区的修改内容，丢弃暂存区的修改内容

示例

首先创建file1.txt、file2.txt、file3.txt三个文件，每次添加一个文件，就commit一次，暂存区中也保存了这三个文件

git reset --soft

附加提交记录的唯一标识即可回退到该版本，提交历史中的file3就消失了，因为回退到file2的版本

工作区file3存在

暂存区file3也还在，依然是跟踪状态

仓库状态新添加了一个文件file3.txt，但是已经被跟踪了，这就是soft的影响，回退的时候，工作区和暂存区的东西不会被清空（暂存区有3个文件)，但是回退到了之前的版本，file3对于file2这个版本就是一个新文件，所以仓库中就没有file3的提交信息了

git reset --hard

HEAD^表示前一个版本

工作区file3不存在了

暂存区file3也没了

仓库状态啥都没了，就是因为hard清空了工作区和暂存区的文件

git reset --mixed

缺省就是mixed模式

工作区file3还在

暂存区file3没了

仓库状态相当于file3是一个新创建且未被跟踪的文件

适用场景

当我们提交了多个版本，但是这么多次没有意义，就可以合并为一个版本，那么就用soft | mixed：

这两者的区别主要在于暂存区还有没有文件，如果所有提交的文件都有用，那么用soft就行，如果有部分是无效的，不需要提交，就用mixed，再手动选择文件进行add，最后提交

忽略文件

.gitignore:忽略掉一些不应该加入到版本库管理的文件，包括

  • 系统或者软件自动生成的文件

  • 编译产生的中间文件和结果文件

    一个文件是通过另外一个文件自动生成的，自动生成的文件没有必要放到版本库中，管理父文件即可，比如java编译生成的.class和.o文件

  • 运行时生成的日志文件、缓存文件、临时文件

  • 涉及身份、密码、口令、密钥等铭感信息文件

分支

介绍

分支（Branch），可以视作代码库中的不同版本，可以独立存在，有自己的提交记录

分支适用于团队协作和开发管理，比如说team1负责开发功能1，team2负责开发功能2，team3负责修复遗留bug，如果都在一个分支上搞，即便修改的是不同的文件，不会冲突，提交记录也会比较混乱，不利于开发。并且，主分支处于不稳定的状态，存在部分功能未开发完成、部分bug未修复等问题。引入分治，主分支随时处于稳定的版本，开发团队可以基于该版本进行功能开发，完善后再合并回来。

解决合并冲突

何为冲突？

两个分支在合并时，如果出现修改的内容重合，则会引发冲突，比如两个分支修改了同一个文件的同一行代码。需要我们手动解决冲突。

  1. 切换回master分支，同样修改master1.txt的内容，并提交，此时master1.txt的原始内容是不包括feat分支所修改的内容的

  1. 现在在master分支上合并feat分支，提示自动合并失败，冲突文件是master1.txt

  1. 使用git status命令查看冲突列表，git diff查看冲突具体内容，这里不显示具体内容是因为windows系统上的txt文件的编码是UTF-8

  1. 手工编辑该文件，留下需要的内容，再添加暂存，最后提交

  2. 如果想要终止此次合并，也可以使用get merge --abort

rebase

中文释义是变基，命令通常是git rebase branch_name，意思是将当前分支变基到branch_name分支上

merge通常是要求在主分支（master或者main）上执行的，其实就是将别的分支合并到主分支，而rebase没有这种限制

执行rebase之后的结果是一条直线，这和merge也是不一样的

实际上，Git中的每个分支都存在一个指针[HEAD]，上述两种不同操作其实就是指针的指向变更不一样

Git会找到这两个分支的最近公共祖先，也就是main3，然后将main3之后当前分支的所有提交移动到目标分支

Rebase vs Merge

Merge的优点在于不会破坏原分支的提交历史，方便回溯和查看，但是会产生额外的提交节点，分支图比较复杂。

Rebase的优点在于不会新增额外的提交记录，形成线性历史，比较直观和干净，但是会改变提交历史（时间顺序会被破坏），避免在共享分支使用，容易造成困扰。

工作流模型

GitFlow

分支被划分为5种类型：

  1. main[主要分支]

    主线分支[长期存在]，项目核心分支，包含项目最新的稳定版本的代码，需要保证该分支的代码随时都是可发布的，能够部署的

    该分支的代码不允许直接修改，只能通过合并别的分支实现版本迭代，每次合并分支都建议生成一个新的版本号，便于追踪和回溯，用git tag命令进行标记，版本号规则：

    1. 主版本（Major Version）：主要的功能变化或重大更新
    2. 次版本（Minor Version）：一些新的功能、改进和更新，通常不会影响现有功能
    3. 修订版本（Patch Version）：一些小的bug修复，安全漏洞补丁等。通常不会更改现有功能和接口。

  2. hotfix[辅助分支]

    问题修复分支[短期存在]，包含项目某个问题修复的源码，用于解决线上问题，所以通常是由main分支分离出来的，修复完成后合并回main分支或者develop分支，并删除该分支，一般命名规则为hotfix-#issueid-desc，通常也会更新小版本号，即修订版本

  3. release[辅助分支]

    预发布分支[短期存在]，包含项目最新预发布版本的代码，用于发布前的测试和验证，通常是从develop分支中分离出来的，当预发布分支的代码稳定后会合并到main分支和develop分支中，并将该预发布分支删除

  4. develop[主要分支]

    开发分支[长期存在]，从main分支分离出来的，包含了项目最新开发版本的代码，用于开发和测试

  5. feature[辅助分支]

    功能分支[短期存在]，包含项目某个新功能的代码，通常是从develop分支中分离出来的。功能开发完成后，合并回develop分支

GithubFlow

GitFlow模型很完善，各个方面都兼备，但是也带来了一定的复杂性，所以就引入GithubFlow，通常包括一个长期存在的主分支[禁止直接提交]和其他功能开发的分支

规范

分支命令

使用带有意义的描述性名称来命名分支

  1. 版本发布应该根据版本号命名：v1.0.0
  2. 功能性分支应该根据其功能和任务命名：feature-login-page
  3. 修补补丁和错误应该根据其问题编号命名：hotfix-#issueid-desc

分支管理

  1. 定期合并已经成功验证的分支，并及时删除已经合并的分支
  2. 保持合适的分支数量
  3. 确保及时将功能性分支合并回主分支
  4. 为分支设置合适的管理权限
