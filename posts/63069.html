计算机网络-应用层

发表于 2019-06-25 | 分类于 课程学习 ， 计算机网络
字数统计: 8.5k | 阅读时长 ≈ 29
计算机网络应用层相关概念，基于中科大郑烇计算机网络课程

概念

只关心应用层自身的功能和传输层提供的服务，屏蔽传输层以下层次的具体细节，在这一层的数据单元叫报文（message）

比如email，Web，网络游戏，流媒体，网络会议，社交网络等等

网络应用的体系结构

网络应用程序的组织结构，描述应用程序在网络中的不同角色和交互方式

客户-服务器模式

  • 服务器

    • 一直运行
    • 固定的IP地址和周知（约定）的端口
    • 扩展性：部署多个服务器，扩展性比较差，达到性能瓶颈后断崖式下降
    • 可靠性：所有资源均位于服务器，可靠性比较差

  • 客户端

    • 主动与服务器通信
    • 与互联网的连接是间歇性的
    • IP地址可能是动态的
    • 不直接与其他客户端通信

对等模式

  • 可以认为没有需要一直运行的服务器
  • 任意端系统之间可以直接进行通信
  • 每个节点可以认为既是服务器也是客户端
  • 自扩展性：新Peer节点带来新的服务能力和服务请求
  • 参与的主机间歇性连接且可以改变IP地址，但是可能带来难以管理的问题
  • 文件分发（bitTorrent）、流媒体、VoIP

混合模式

Napster

  • 文件搜索：C/S
    • 任意端系统向服务器注册自己拥有的资源
    • 任意端系统在服务器检索所需资源的位置
  • 文件传输：P2P
    • 任意端系统之间都可以进行数据传输

即时通讯

  • 在线检测：C/S
    • 用户上线时，向中心服务器注册其IP地址
    • 用户与中心服务器联系，找到在线好友的位置
  • 聊天：P2P

进程通信

  • 进程：在主机上运行的应用程序
  • 同一个主机：使用进程间通信机制（操作系统）进行通信
  • 不同主机：通过交换报文（Message）进行通信
    • 需要使用操作系统提供的进程间通信服务
    • 借助传输层提供的服务，按照应用协议交换报文

分布式进程通信

进程标识和寻址问题

进程为了接收正确的报文，需要有一个标识（IP + port），唯一代表主机上的这个进程

  1. 主机：唯一的32位IP地址
  2. 端口号（Port Number）
  3. 需要使用到的传输层协议TCP UDP

应用层功能

应用层协议定义了应用程序与网络之间的通信规则和数据格式，使得不同设备和应用程序能够互相协作和交换数据。协议是应用层的一个组成部分，例如web应用由HTTP协议、web浏览器、web服务器和HTML协议等部分组成

通信规则和数据格式

  1. 交换的报文类型：请求报文和应答报文
  2. 不同报文类型的语法规范：报文中的应该有哪些字段，分别有什么作用
  3. 字段的语义：字段取某个值有什么含义
  4. 进程什么时候，怎么样发送报文，以及对报文如何进行响应的规则

主要功能

  1. 提供应用程序接口（API）：应用层协议定义了应用程序与网络之间的接口，使得应用程序可以方便地访问网络资源和服务。常见的应用层协议包括HTTP、FTP、SMTP等，它们提供了不同的API，用于实现Web服务、文件传输、电子邮件等功能。
  2. 实现应用层协议：应用层协议定义了应用程序与网络之间的通信规则和数据格式，应用程序必须遵守这些规则才能与网络进行通信。例如，HTTP协议定义了Web浏览器和Web服务器之间的通信规则，包括请求和响应的格式、请求方法、状态码等，使得Web浏览器和Web服务器能够互相通信和交换数据。
  3. 支持数据编码和解码：在网络中传输的数据需要进行编码和解码，以确保数据的正确性和完整性。应用层协议可以支持不同的数据编码和解码方式，例如Base64编码、压缩编码等，以便于在网络中传输和处理数据。

为了实现上述的应用层功能，需要借助传输层的服务，也就需要将数据进行层间垂直传输

数据：

  • 要传输的报文（本层：服务数据单元SDU）
  • 源：自身的IP和TCP（UDP）端口号
  • 目的地：对方的IP和TCP（UDP）端口号

socket

每次层间数据传输，除了服务数据单元外，都要传这么多上述数据（源IP 目标IP等等），过于繁琐，容易出错，也难以管理，引入socket这一个唯一的整数用来在本地唯一标识通信的双方（TCP）或者单方（UDP），作用范围是本地

类比操作系统中的文件句柄，使用这个句柄，就可以操作文件，不需要每次都获取文件的目录、文件名及打开方式，或者C++中的指针

TCP socket

TCP服务：两个进程之间的通信需要预先建立连接，通信一般会持续一段时间，通信关系比较稳定，比如文件传输

socket：用一个唯一的整数标识具有本地意义的四元组<源IP，源端口，目标IP，目标端口>，唯一指定了一个对话（两个进程之间的对话）

特点：

  1. 通信时，不需要发送每个报文都附带这个四元组，通过socket就可以标识
  2. 简单，易于管理
UDP socket

UDP服务：两个进程之间的通信无需建立连接，每个报文都是独立传输的，不关心是否发送/接收成功，本地进程发送的前后报文可能给不同的进程（目标IP和目标端口不一样）

socket：用一个唯一的整数标识具有本地意义的二元组（源IP，源端口），目标IP和目标端口需要提供，唯一指定了一个本地的应用进程

特点：

  1. 在发送数据报时，采用创建好的本地套接字，就不必在发送每个报文中指明自己所采用的ip和port
  2. 发送报文时，还是要指定对方的ip和udp 端口号

传输层服务

传输层实体（tcp或者udp实体）根据上述数据进行TCP报文段或者UDP数据报的封装，进一步将IP地址往下送给网络层的IP实体，进行IP数据包的封装

服务指标

  • 数据丢失率

    • 有些应用则要求100%的可靠数据传输（如文件）

    • 有些应用（如音频）能容忍一定比例以下的数据丢失

  • 延迟

    • 一些应用 出于有效性考虑，对数据传输有严格的时间限制

    • Internet电话、交互式游戏

  • 吞吐

    • 一些应用（如多媒体）必须需要最小限度的吞吐（要不然看视频一卡一卡的），从而使得应用能够有效运转
    • 一些应用能充分利用可供使用的吞吐(弹性应用)

  • 安全性

    • 机密性
    • 完整性
    • 可认证性

TCP服务
  1. 可靠的传输服务
  2. 流量控制：发送方不会淹没接收方
  3. 拥塞控制：当网络出现拥塞时，能抑制发送方的发送速率
  4. 面向连接：两个进程之间需要建立连接才能通信
  5. 无法保证延迟、吞吐量和安全性
UDP服务
  1. 不可靠的传输服务
  2. 无需建立连接
  3. 无法提供流量控制、拥塞控制、延迟、带宽、安全性等

必要性：

  1. 网络层的IP不能区分进程，需要用到UDP 端口号区分进程

  2. 无需建立连接，节省时间

    比较适合事务性的应用，比如

    1. 域名系统（Domain Name System，DNS）：DNS是一种将域名映射到IP地址的分布式数据库系统，它使用UDP协议进行域名解析和查询，由于查询速度要求高、数据量小，因此使用UDP协议更加合适。
    2. 动态主机配置协议（Dynamic Host Configuration Protocol，DHCP）：用于自动分配IP地址、子网掩码、网关等网络参数，它使用UDP协议实现客户端与服务器之间的通信，由于数据量小、传输速度要求高，因此使用UDP协议更加合适。
    3. 基于IP的语音传输（Voice over Internet Protocol，VoIP）：VoIP是一种基于网络的语音通信技术，它使用UDP协议进行语音数据的传输，由于实时性要求高、数据量小、传输速度要求高，因此使用UDP协议更加合适。

  3. 不用付出时间代价做可靠性的工作

    比如检错重发、确保有序等，实现简单，比较适合对实时性要求较高的应用，比如流媒体

  4. 没有拥塞控制和流量控制，应用能够按照预设的速度发送数据

    TCP上面的应用发送数据的速度和主机向网络发送的实际速度是不一致的，因为有流量控制和拥塞控制

Web and HTTP

相关概念

  • 万维网（World Wide Web, WWW）:分为 Web 客户端和 Web 服务器程序。WWW 可以让 Web 客户端（常用浏览器）访问浏览 Web 服务器上的页面。是一个由许多互相链接的超文本组成的系统，通过互联网访问。在这个系统中，每个有用的事物，称为一样“资源”；并且由一个全局“统一资源标识符”（URI）标识；这些资源通过超文本传输协议（Hypertext Transfer Protocol）传送给用户，而后者通过点击链接来获得资源。万维网实际上是运行在互联网上的一种应用服务，提供了互联网上的信息内容
    • 互联网定义了网络间通信的规则,而万维网定义了在互联网上传输内容的规则
    • 互联网是网络的集合,让计算机能够相互通信;万维网是一个信息服务,提供互联网上的内容
    • 互联网存在于物理层,万维网存在于应用层。
    • 互联网把网络相连在一起,而万维网为这些网络中的计算机和设备提供信息服务。
  • 超文本标记语言（Hypertext Markup Language，HTML）：一种基于文本的语言，用于描述网页的结构和内容，通过标签（tag）和属性（attribute）来定义网页的元素和样式
  • 统一资源定位符（URL）：统一资源定位符是对可以从互联网上得到的资源的位置和访问方法的一种简洁的表示，是互联网上标识资源的地址。互联网上的每个文件都有一个唯一的 URL，它包含的信息指出文件的位置以及浏览器应该怎么处理它。
  • Web页：由一些对象构成
    • 对象可以是HTML文件、JPEG图像、Java小程序、声音剪辑文件等
    • Web页首先含有一个基本的HTML文件，这个文件又包含若干对象的引用（链接）
    • 通过统一资源定位符（Uniform Resource Locator，URL）对每个对象进行引用
      • 访问协议，用户名，口令字，端口等
  • 往返时间（Round Trip Time，RTT）：一个小的分组从客户端到服务器，再回到客户端的时间（传输时间忽略）
  • 响应时间：客户端发起请求后得到响应的时间

超文本传输协议

超文本传输协议（Hypertext Transfer Protocol，HTTP）：一种用于在Web浏览器和Web服务器之间基于TCP协议交换数据的应用层协议

属于客户/服务器模式，客户是请求、接收和显示Web对象的浏览器，服务器是对请求进行响应，返回对象的服务器

基于TCP协议实现HTTP协议

  1. 服务器启动，初始化一个守护socket，在某个固定IP的80号端口上保持监听状态

    守护socket，不代表任何会话关系，在服务器端保存持久连接，起到监听的作用

  2. 客户发起一个与服务器的TCP连接（80号端口）

  3. 服务器接受客户的TCP连接，生成一个新的socket，代表服务器与这个客户端的会话关系

  4. 在浏览器(HTTP客户端)与 Web服务器(HTTP服务器 server)交换HTTP报文 (应用层协议报文)

    浏览器收到返回的HTTP报文后，需要绘制出来，通过URL进行对象解析，对链接、图片都要这样操作，获取资源进行绘制

  5. TCP连接关闭

HTTP连接

无状态的连接：服务器并不维护任何关于客户的信息

因为维护状态的协议很复杂：

  1. 必须维护历史信息
  2. 如果服务器/客户机死机，可能导致双方的状态信息不一致，实现同步很困难
  3. 无状态的服务器可扩展性更好，能够支持更多客户端的连接

非持久HTTP（HTTP/1.0时代）

  • 最多只有一个对象在TCP连接上发送， 下载多个对象需要建立多个TCP连接

    假设用户输入URL：http://www.someschool.edu/someDept/home.index

    1. HTTP客户端在端口号80发起一个到服务器www.someSchool.edu的连接
    2. 位于主机www.someSchool.edu的HTTP服务器在80号端口等待连接，接受连接并通知客户端
    3. HTTP客户端向TCP连接的套接字发送HTTP请求报文，报文表示客户端需要对象someDepartment/home.index
    4. HTTP服务器接收到请求报文，检索出被请求的对象，将对象封装在一个响应报文，并通过其套接字向客户端发送
    5. HTTP关闭TCP连接
    6. HTTP客户端收到包含html文件的响应报文，并显示html。然后对html文件进行检查，找到10个引用对象
    7. 对10个引用对象，重复上述1-7的操作，直到所有引用对象都绘制出来

    响应时间=2RTT+文件传输时间：一个RTT用来发起和建立TCP连接（这里传输的数据很少，传输时间忽略不计），一个RTT用来发送HTTP请求并等待HTTP响应，最后还有文件传输时间

    缺陷

    • 每个对象传输都要2个RTT用来建立连接和回应请求
    • 操作系统需要为每个对象的每个TCP连接分配资源，而浏览器通常都会打开并行TCP连接获取引用的资源

    请求方法类型

    • GET：请求资源
    • POST：提交表单
    • HEAD：要求服务器在响应报文中不包含请求对象 故障跟踪

持久HTTP（HTTP/1.1时代）

  • 多个对象可以在一个TCP连接（客户端-服务器）上进行传输，其实就是连接不关闭，后续有数据要传输，直接传输

  • 服务器发送HTTP报文后不关闭连接，保持响应

  • 在相同客户端和服务器之间的后续请求和响应报文通过相同的连接进行传送

  • 客户端在遇到一个引用对象的时候，就可以尽快发送该对象的请求

    非流水线的持久HTTP：

    • 客户端只能在收到前一个响应后才能发出新的请求
    • 每个引用对象花费一个RTT

    流水线的持久HTTP：

    • 客户端遇到一个引用对象就立即产生一个请求，不需要等待前一个请求的回应
    • 所有引用对象在最理想的情况下可能只需要一个RTT

    请求方法类型

    1. GET,POST,HEAD

    2. PUT：将实体主体中的文件上载到URL字段规定的路径

    3. DELETE：删除URL字段规定的文件

      后两个一般是网络管理员用于维护网页

HTTP请求报文

请求报文

提交表单输入

网页通常包含表单的输入

  1. Post方式

    包含在实体主体（entity body）中的输入被提交到服务器

  2. Get方式（URL的方式）

    输入通过请求的URL字段上载

HTTP响应报文

格式

状态码

200 OK：请求成功，请求对象包含在响应报文的后续部分

301 Moved Permanently：请求的对象已经被永久转移了；新的URL在响应报文的Location:首部行中指定，客户端软件自动用新的URL去获取对象

400 Bad Request：一个通用的差错代码，表示该请求不能被服务器解读

404 Not Found：请求的文档在该服务上没有找到

Cookies

用于维护用户-服务器的连接状态， 在四个地方记录或者使用

  1. 在HTTP响应报文中有一个cookie的首部行，使用
  2. 在HTTP请求报文含有一个cookie的首部行，使用
  3. 在用户端系统中保留有一个cookie文件，由用户的浏览器管理，记录
  4. 在Web站点有一个后端数据库，记录

后续访问，客户端带上这个cookie，服务器识别这个cookie就知道怎么回应了（购物车信息，产品推荐…）

便利性：

  1. 用户验证
  2. 购物车记录
  3. 推荐
  4. 用户状态
Web缓存

代理服务器（Proxy Server）：代理服务器（Proxy Server）是一种网络实体，它又称为万维网高速缓存。 代理服务器把最近的一些请求和响应暂存在本地磁盘中。当新请求到达时，若代理服务器发现这个请求与暂时存放的的请求相同，就返回暂存的响应，而不需要按 URL 的地址再次去互联网访问该资源。代理服务器可在客户端或服务器工作，也可以在中间系统工作。

代理服务器（大学、公司、居民区的ISP）设置缓存，其实就是客户不需要访问原始服务器，由代理服务器满足客户的需求

实际上，客户端自身也能进行缓存

用户设置浏览器：运行通过代理服务器的缓存访问网页

浏览器将所有的HTTP请求发给代理服务器

  1. 缓存中的对象直接返回
  2. 不在缓存中的对象，代理服务器去向原始服务器要，存储到缓存中，再返回给客户端

优势

  1. 降低客户端响应时间
  2. 大大减少一个区域内部和互联网这个外部的交互流量
  3. 降低原始服务器的负担

条件式获取（Conditional GET）：访问的对象实际上在服务器端已经更新了

  1. 用户向代理服务器请求某个对象，这个对象在代理中有，但是不确定是不是最新的
  2. 在代理向服务器的请求报文中添加头部 If-modified-since: ，实际上这个日期应该是最近的一次缓存该对象的日期 ，询问这个对象自从这个日期后有没有进行修改
  3. 如果没有修改，那么返回304 Not Modified
  4. 如果有修改，那就需要服务器重传一份给代理

文件传输协议

文件传输协议（File Transfer Protocol，FTP）用于在本地文件系统和远程文件系统之间传输文件，基于TCP实现，同时FTP也是一个应用程序

客户/服务器模式

  • 客户：发起传输的一方
  • 服务器：远程主机

客户端和服务器通过端口21进行连接，并使用TCP作为传输协议

控制连接（control connection）：用于传输FTP命令（登录、列出目录、上传文件等）和响应信息（成功、失败、文件已存在），以便客户端和服务器进行通信，控制连接始终由客户端发起，始终保持连接状态直到客户端关闭连接或者服务器超时而关闭

数据连接（data connection）：用于传输文件内容的连接

控制连接其实是带外（out of band）传输的体现，带外传输指:使用与主要数据流（in-band，带内）不同的通信通道传输控制信息或数据的过程

执行流程

  1. 客户端通过控制连接获得身份确认（身份认证，明文传输）
  2. 客户端通过控制连接发送命令浏览远程目录
  3. 服务器收到一个文件传输命令时，打开一个与客户端的数据连接
  4. 文件传输完成后，数据连接关闭，但是控制连接依然保持
  5. 如果有新的文件需要传输，服务器会打开新的数据连接进行传输

FTP服务器维护用户的状态信息：当前访问了哪个目录，用户账户与控制连接进行对应

电子邮件

主要组成部分

  • 用户代理
  • 邮件服务器
  • 简单邮件传输协议：SMTP

用户代理

邮件阅读器：Outlook、Foxmail

  • 负责撰写、编辑和阅读邮件
  • 输出和输入邮件保存在服务器上

邮件服务器

  • 邮箱：管理和维护发送给用户的邮件
  • 待发送：以报文队列保存待发送的邮件报文
  • 邮件服务器之间利用SMTP协议互发email报文
  • 用户发邮件：使用TCP协议基于25号端口将邮件发给服务器，服务器基于TCP协议投递到对方在服务器中的邮箱
  • 用户收邮件：从服务器中自己的邮箱中拉取邮件，需要使用POP3 IMAP和HTTP协议
    • 邮局访问协议 第三版（Post Office Protocol version 3，POP3）
      1. 只能区分 收件箱和发件箱
      2. 想要阅读邮件只能下载下来，无法在服务器上整理邮件
      3. 一次只能从一个设备访问邮件
      4. 不支持在线搜索邮件内容
    • Internet 邮件访问协议（Internet Mail Access Protocol，IMAP）
      1. 允许在远程的服务器上创建目录，移动文件
      2. 允许多个设备同时访问邮件
      3. 下载前可以阅读部分邮件内容
      4. 支持在线搜索邮件内容

示例

  1. Alice使用用户代理撰写邮件并发送给bob@someschool.edu
  2. Alice的用户代理将邮件发送到她的邮件服务器；邮件放在报文队列中
  3. SMTP的客户端打开到Bob邮件服务器的TCP连接
  4. SMTP客户端通过TCP连接发送Alice的邮件
  5. Bob的邮件服务器将邮件放到Bob的邮箱
  6. Bob调用他的用户代理阅读邮件

SMTP协议

一种用于发送电子邮件的标准协议，它定义了如何将电子邮件从一个邮件客户端发送到一个或多个邮件服务器（下一跳是哪），并最终发送到目标邮件服务器上的收件人的电子邮件客户（只负责发送，不负责接收，接收由POP3 IMAP实现）

作用范围

  1. 客户端通过SMTP服务发送给邮箱服务器
  2. 邮箱服务器之间通过SMTP服务互相传输邮件

工作流程

  1. 邮件客户端向邮件服务器发送连接请求。
  2. 邮件服务器响应连接请求，并等待邮件客户端发送邮件。
  3. 邮件客户端发送邮件到邮件服务器，包括邮件的主题、正文和附件等信息。
  4. 邮件服务器接收邮件，并将其存储在邮件队列中，等待发送到目标邮件服务器。
  5. 邮件服务器使用DNS来查找目标邮件服务器的地址。
  6. 邮件服务器向目标邮件服务器发送邮件。
  7. 目标邮件服务器接收邮件，并将其存储在邮件队列中，等待目标邮件客户端检索。
  8. 目标邮件客户端从邮件服务器中检索邮件，并将其显示在邮件客户端中。

域名系统

必要性

IP地址虽然能够标识主机、路由器，但是不好记忆，需要用字符串的名称来标识IP地址，进一步就需要一个服务能够负责将名称转换为IP地址

Domain Name System，DNS：一个由分层的DNS服务器实现的分布式数据库系统，同时也是一个帮助主机查询这个分布式系统的基于UDP（53）实现传输的协议，实际上就是为互联网设计的电话簿

  1. 分层的、基于域的命名机制
  2. 若干分布式的数据库完成名字到IP地址的转换

术语解释

  1. 顶级域（Top Level Domain，TLD）DNS服务器：负责顶级域名（如com, org, net, edu和gov）和所有国家级的顶级域名（如cn, uk, fr, ca, jp）
  2. 权威DNS服务器：组织机构的DNS服务器， 提供组织机构服务器（如Web和mail）可访问的主机和IP之间的映射
  3. 根DNS服务器：最顶层的DNS服务器，知道所有顶级域服务器的域名
  4. 本地DNS服务器：不严格属于层次结构，默认名字服务器，代理作用，比较近

主要目的

  1. 主机名和IP地址的转换
  2. 邮件服务器别名到邮件服务器正规名字的转换

域名结构

  • 层次树状结构的命名方法
  • Internet根被划分为几百个顶级域（Top Level Domain，TLD）
    • 通用的域：.com .edu .gov .net
    • 国家的域：.cn .us
  • 每个域下面都可划分为若干子域（subdomains）
  • 叶子节点是主机

命名空间

  • 域名结构

    • 从本域相似，直到树根，中间用.间隔不同的级别

      ustc.edu.cn

    • 域的域名：用来表示一个域

    • 主机的域名：表示域内的一个主机

  • 域名管理

    • 一个域管理其下所有域

      .cn 被划分为 edu.cn gov.cn

    • 创建域需要征得父域的同意

    • 域与物理网络无关，逻辑划分

      域内主机可以不在一个网络，一个网络内的主机可以不在一个域

解析服务

将命名空间划分为互不相交的区域，每个区域都是树的一部分，每个区域都有自己的名字服务器，维护辖区内的权威信息（authoritative record），名字服务器允许放置于区域外，以保障可靠性

资源记录

Resource Records

  • 维护 域名-IP地址的映射关系，位于名字服务器的分布式数据库中

    格式：（domain_name，ttl，type，class，value）

    1. domain_name：域名

    2. ttl：time to live生存时间，决定资源记录从缓存中删除的时间，权威的生存时间无限长，缓冲记录则存在有效期

    3. class：类

    4. type：类型，资源记录是什么类型，区分这个映射关系的作用

      1. A：主机->IP地址

      2. CNAME：别名->正规名称

        www.ibm.com的规范名称为servereast.backup2.ibm.com

      3. NS：域名->这个域的权威服务器的域名

      4. MX：名称->邮件服务器的名字

    5. value：值，IP地址、域名、等等

工作过程

  1. 应用调用解析器（resolver）

  2. 解析器作为客户 向Name Server发出查询报文，封装到UDP数据报中

  3. Name Server进行名字解析，然后返回响应报文

    1. 目标名字在本地名字服务器中，直接返回

      1. 查询的名字就在区域内部

      2. 有缓存

        提高性能，根服务器通常都在本地服务器有缓存，使根服务器不需要经常被访问，不一致问题通过TTL来解决

    2. 不在本地名字服务器中，联系根名字服务器，要找到权威名字服务器

      1. 递归查询

        根服务器需要接收查询请求2，亲自去查3 6，后续还得把结果返回7

      2. 迭代查询

        根服务器只需要告知，你去哪里可能可以查到这个域名，只需要2 3操作

本地名字服务器

每个ISP都有一个本地DNS服务器，或者叫默认名字服务器，代理作用，距离比较近，查询比较方便

域名维护

如何新增一个域？

需要在上级域的名字服务器中新增两项内容

  1. 新增的子域的域名=>子域能在哪个名字服务器找到，名字服务器的名字
  2. 名字服务器=>服务器IP地址

比如新增networkutopia.com，对应的权威DNS服务器是dns1.networkutopia.com，IP地址是212.212.212.1

  1. (networkutopia.com, dns1.networkutopia.com）
  2. (dns1.networkutopia.com, 212.212.212.1)

在dns1.networkutopia.com必须保存有一个类型A和MX的记录

  2. 邮件服务

对等应用

纯P2P架构

相较于C/S架构，P2P在文件传输上存在天然优势

C/S架构的传输耗时随着客户端增加而线性增加，P2P随着客户端增加缓慢增加（客户端在带来下载需求时，也提供了服务能力）

tracker：跟踪torrent中的参与节点，某个文件有哪些客户在上传和下载，peer可以同时向拥有该文件的peer进行下载（文件被划分为多个块，向不同的peer申请不同的块，并发下载）

torrent：节点的组，组内节点间互相交换文件块

churn：扰动，peer节点可能会上线、下线

传输原则

并非先来先到，而是tit-for-tat，一报还一报，谁对我好，我就对谁好，自己提供的带宽越大，别人也愿意提供服务

多媒体服务

多媒体流化技术

Dynamic, Adaptive Streaming over HTTP

服务器：

  • 将视频文件分割成多个块，比如8-10s一块
  • 每个块独立存储，编码于不同码率，解析度，或者说清晰度
  • 告示文件（manifest file）：提供不同块的URL，存储在哪里，从哪里可以获取这个视频

客户端：

  • 先获取告示文件
  • 周期性测量服务器到客户端的带宽
  • 查询告示文件,在一个时刻请求一个块，HTTP头部指定字节范围
    • 如果带宽足够，选择最大码率的视频块
    • 会话中的不同时刻，可以切换请求不同的编码块 (取决于当时的可用带宽)
  • 动态自适应客户端
    • 什么时候去请求块 (不至于缓存挨饿，或者溢出)
    • 请求什么编码速率的视频块 (当带宽够用时，请求高质量的视频块)
    • 去哪里请求块 (可以向离自己近的服务器发送URL，或者向高可用带宽的服务器请求)

内容分布式网络

Content Distribution Network，CDN

面临规模性（数以亿计的用户）、异构性（不同用户的带宽、延迟不同）的挑战，集中式不可取

通过CDN，全网部署缓存节点，存储服务内容，就近（网络路径拥塞，换一条路）为用户提供服务

动态主机配置协议

除了系统管理员静态配置IP地址，还有动态配置

动态主机配置协议（Dynamic Host Configuration Protocol，DHCP）：基于UDP的一个服务协议，允许主机从服务器动态获取一个IP地址，并包括子网掩码、本地域名服务器（域名和IP地址）、默认网关（第一跳路由器的IP地址）等。该协议允许主机在加入网络时，动态地从服务器获取IP地址，简化网络管理，提高网络配置的灵活性和可扩展性

互联网名字与编号分配机构（ Internet Corporation for Assigned Names and Numbers，ICANN）:负责管理Internet的全球域名系统（DNS）和IP地址分配

假设现在总共有400台主机，IP地址只能支持256个，是否可以不增加IP地址支持400个主机呢？

400个最多的时候也不会有256个同时上网，所以可以动态分配，这台主机不上网了，IP地址就空出来了

  1. 更新主机的IP地址，IP地址是有租期的，到期了需要进行更新
  2. 主机重启时，允许主机使用以前用过的IP地址
  3. 支持移动用户入网（短期在网）

其他应用或者协议

简单文件传输协议（TFTP）：TFTP（Trivial File Transfer Protocol,简单文件传输协议）是 TCP/IP 协议族中的一个用来在客户机与服务器之间进行简单文件传输的协议，提供不复杂、开销不大的文件传输服务。端口号为 69。

远程终端协议（TELNET）：Telnet 协议是 TCP/IP 协议族中的一员，是 Internet 远程登陆服务的标准协议和主要方式。它为用户提供了在本地计算机上完成远程主机工作的能力。在终端使用者的电脑上使用 telnet 程序，用它连接到服务器。终端使用者可以在 telnet 程序中输入命令，这些命令会在服务器上运行，就像直接在服务器的控制台上输入一样。可以在本地就能控制服务器。要开始一个 telnet 会话，必须输入用户名和密码来登录服务器。Telnet 是常用的远程控制 Web 服务器的方法。
