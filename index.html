git本地安装、配置和使用

安装

Git - 安装 Git (git-scm.com)

安装完成后，即可在终端输入git -version命令，如果能够显示版本信息，表示安装成功

使用方式

  1. 命令行

  2. 图形化界面（Sourcetree、fork、Github Desktop）

    Git - GUI Clients (git-scm.com)

  3. IDE插件/扩展

    在常用的IDEA或者VSCode这些IDE工具中，通过插件或者扩展的方式使用git

配置

配置用户名和邮箱，识别是谁提交的内容（首次执行即可）

--global 表示全局配置，对所有仓库都生效

--system 表示系统配置，对所有用户都生效

缺省（local）表示本地配置，只对本地仓库有效

使用上述命令保存用户名和密码，避免每次都输入，这个命令设置了一个全局git配置，相当于告诉git在需要身份验证时使用store的凭证助手

使用上述命令可以查看我们配置的用户信息

使用

新建版本库

repository（repo）可以理解为一个目录，目录内所有文件都可以被Gti管理起来，对文件的任意操作（修改、删除、添加）都能被Git跟踪到，以便追溯或者还原任意文件到某个历史版本

本地创建

直接在本地某个目录执行git init

.git文件夹存储了Git仓库的所有数据，删除该文件，仓库也就被删除了

git init my-repo则会在当前目录中创建一个子目录my-repo作为仓库

远程创建

通过git clone从Git的远程服务器克隆一个已经存在的仓库

工作区域和文件状态

Git的本地数据管理分为三个区域：

  1. 工作区：电脑上资源管理器能看到的目录、文件（.git所在的目录，可以进行操作）
  2. 暂存区：临时存储区域，用于保存即将提交到Git仓库的修改内容，用于进行版本控制（中间区域，临时存放内容）
  3. 本地仓库：存储完整的项目历史和元数据，是Git存储代码和版本信息的主要位置

首先通过git add将被修改的文件添加到暂存区，然后通过git commit将文件内容统一提交到本地仓库，在这个过程中可以使用相关命令进行查看、比较或者撤销修改

工作区的文件也分为四种状态：

  1. 未跟踪：新创建的文件，还没有被Git管理起来的文件
  2. 未修改：已被管理但文件的内容没有修改
  3. 已修改：已被管理且文件内容有所修改
  4. 已暂存：已经添加到暂存区中的文件

添加和提交文件

git status

查看当前仓库的状态，包括处于哪个分支、存储了哪些文件、文件状态如何

创建一个新文件 test.txt，再次查看仓库状态

新的文件test.txt目前就是未跟踪的状态（红色）

git add

常规使用

通过git add test.txt将该文件添加到暂存区，如果是新文件，那么就是对其进行跟踪

新文件也就被跟踪了（绿色），仓库中存在已经被修改但是没有commit的文件，同时用git rm --cached <file>这个命令可以将暂存区的文件取消暂存，这样文件又变回未跟踪状态了

通配符

git add还可以使用通配符*，比如我们创建test2.txt，test3.txt及test4.md

git add *.txt就可以把所有以.txt结尾的文件都添加到暂存区

目录

使用git add .之后的仓库状态，就不存在未被跟踪的文件了

git commit

通过git commit将所有暂存区中的文件进行提交，与工作区中的文件无关，比如再创建一个新文件test1.txt

需要注意的是git commit需要-m来设置提交的信息，该信息也会被记录到仓库中，提交完后，未被提交的文件已经消失了，只剩下未被跟踪的test2.txt

如果未使用-m指定提交信息，那么就会进入交互式界面，通过vim编辑提交信息（i进入插入模式，esc回到命令模式，输入:wq保存退出），此时所有文件就都提交了

git log

通过该命令可以查看提交历史记录，包括comit后面这个16进制的字符串就是唯一ID、提交作者、日期及提交信息

git log --oneline显示简洁的提交记录

回退版本

git reset

退回到之前某一个提交commit的状态，有三种用法

  1. git reset --soft：回退到某一个版本，并且保留工作区和暂存区的所有修改内容
  2. git reset --hard：回退到某一个版本，但是丢弃工作区和暂存区的所有修改内容
  3. git reset --mixed（默认参数）：回退到某一个版本，并且只保留工作区的修改内容，丢弃暂存区的修改内容

示例

首先创建file1.txt、file2.txt、file3.txt三个文件，每次添加一个文件，就commit一次，暂存区中也保存了这三个文件

git reset --soft

附加提交记录的唯一标识即可回退到该版本，提交历史中的file3就消失了，因为回退到file2的版本

工作区file3存在

暂存区file3也还在，依然是跟踪状态

仓库状态新添加了一个文件file3.txt，但是已经被跟踪了，这就是soft的影响，回退的时候，工作区和暂存区的东西不会被清空（暂存区有3个文件)，但是回退到了之前的版本，file3对于file2这个版本就是一个新文件，所以仓库中就没有file3的提交信息了

git reset --hard

HEAD^表示前一个版本

工作区file3不存在了

暂存区file3也没了

仓库状态啥都没了，就是因为hard清空了工作区和暂存区的文件

git reset --mixed

缺省就是mixed模式

工作区file3还在

暂存区file3没了

仓库状态相当于file3是一个新创建且未被跟踪的文件

适用场景

当我们提交了多个版本，但是这么多次没有意义，就可以合并为一个版本，那么就用soft | mixed：

这两者的区别主要在于暂存区还有没有文件，如果所有提交的文件都有用，那么用soft就行，如果有部分是无效的，不需要提交，就用mixed，再手动选择文件进行add，最后提交

hard误删

git reflog查看所有操作历史记录，再使用reset进行回溯即可
